## 1. Документообіг

### 1.1 Життєвий цикл документів ✅ — в цілому правильно

| Документ | Статуси | Коректність |
|----------|---------|-------------|
| Прихід (receipt) | draft → confirmed → cancelled | ✅ |
| Заявка (order) | draft → submitted → confirmed → partially_shipped → shipped → cancelled | ✅ |
| Відвантаження (shipment) | draft → packed → shipped → delivered → cancelled | ⚠️ див. нижче |
| Переміщення (transfer) | draft → confirmed → completed → cancelled | ✅ |
| Списання (write-off) | draft → confirmed → cancelled | ✅ |
| Інвентаризація (inventory) | draft → in_progress → completed → cancelled | ✅ |

**Зауваження:**

#### 1.1.1 🟡 Shipment — статус `delivered` не використовується
У коді `ship_order()` статус одразу ставиться `shipped`. Статус `delivered` ніколи не встановлюється. Для внутрішнього переміщення між складами мережі доставка — критичний етап. Машина виїхала ≠ товар отримано.

**Рекомендація:** Додати RPC `deliver_shipment()` або автоматичне підтвердження отримання через Telegram-бота.

#### 1.1.2 🟡 Часткове відвантаження — `partially_shipped` є, але недопрацьовано
Статус `partially_shipped` присутній у CHECK orders.status, але в коді:
- `ship_order()` створює ВСЕ відвантаження одразу — або всі товари, або нічого
- Немає механізму "відвантажити тільки X з Y"
- У таблиці `order_items.quantity_shipped` є, але не використовується в бізнес-логіці

**Рекомендація:** Доробити часткове відвантаження — оператор обирає скільки одиниць кожного товару відвантажити, замість усього замовлення.

#### 1.1.3 🟡 Відвантаження не прив'язане до накладної
У реальному складі: заявка → накладна на збірку → відвантаження. Зараз `ship_order()` створює `shipments` та одразу зменшує залишки без створення документа-підстави (видаткова накладна).

**Рекомендація:** Додати видаткову накладну (expense_note / issue_note) — окремий документ між заявкою та відвантаженням для юридичної значущості.

#### 1.1.4 🟢 Прихід — зроблено добре
- `rpc_create_receipt_with_items` — атомарне створення (заголовок + рядки в одній транзакції) ✅
- `confirm_receipt` з `FOR UPDATE` lock ✅
- Номер документа через `next_document_number` ✅

### 1.2 Нумерація документів

#### 1.2.1 ✅ `next_document_number` — правильне рішення
Атомарний INSERT...RETURNING + year + prefix — закриває race condition на опівночі. Префікси ORD, RCP, TRN, WRT, SHp.

#### 1.2.2 🟡 Немає префіксу для інвентаризації та списань
`next_document_number` підтримує будь-який префікс, але в коді не всі документи його використовують:
- `confirm_write_off` — не викликає `next_document_number` (бере `write_offs.transfer_number`)
- `complete_inventory` — не генерує номер

**Рекомендація:** Всі документи повинні мати унікальний номер через `next_document_number`.

## 2. Складський облік

### 2.1 Stock Engine

#### 2.1.1 ✅ `update_stock_balance` — атомарний UPSERT
Використовує `ON CONFLICT DO UPDATE SET quantity = sb.quantity + EXCLUDED.quantity` — правильно, без race condition.

#### 2.1.2 ✅ Журнал рухів (stock_movements) — immutable
Кожна зміна залишку записується в `stock_movements` з типом руху, референсом, користувачем. Аудит залишків — коректно.

#### 2.1.3 🟡 Немає перерахунку залишків (reconciliation)
Якщо з якоїсь причини stock_balances розійдуться з stock_movements (баг, ручне втручання, збій), немає функції перерахунку "з нуля" за історією рухів.

**Рекомендація:** Додати RPC `reconcile_stock(warehouse_id)` — перераховує stock_balances як суму всіх stock_movements з початковим залишком.

#### 2.1.4 🔴 Прихід не фіксує ціну в залишках
`confirm_receipt` додає кількість у `stock_balances`, але не зберігає ціну надходження. `stock_balances` не має колонки `price`. При списанні немає можливості розрахувати собівартість (FIFO/середня).

**Рекомендація:** Додати:
- `stock_balances.unit_price NUMERIC(12,2)` — середньозважена ціна
- При оприбуткуванні перераховувати: `(old_qty * old_price + new_qty * new_price) / (old_qty + new_qty)`
- `stock_movements.unit_price NUMERIC(12,2)` — ціна на момент руху

Це дасть змогу:
- Бачити вартість залишків (зараз `stock_value` у дашборді — добуток `quantity * purchase_price`, а purchase_price може змінитися)
- Розраховувати собівартість списання
- Фінансовий аналіз

### 2.2 Мінімальні залишки (min_stock)

#### 2.2.1 🟡 min_stock є, але не автоматизовано
Зараз `min_stock` — це просто число в таблиці `products`. Немає:
- Автоматичного перерахунку min_stock на основі середньоденного споживання
- Формування заявки постачальнику при досягненні min_stock
- Cповіщення у Telegram при критичному залишку

**Рекомендація:** 
- Додати RPC `calculate_reorder_point(product_id, days_cover)` — аналізує витрату за N днів, пропонує новий min_stock
- Додати фонове завдання (pg_cron) — щодня перевіряє stock_balances, відправляє сповіщення в Telegram при досягненні порогу
- Додати автоформування заявки постачальнику на основі товарів нижче min_stock

### 2.3 Складська ієрархія

#### 2.3.1 🟡 Типи складів — дублювання
У `warehouses` є дві колонки з типами:
- `type TEXT CHECK (type IN ('central', 'shop', 'transit'))` — з оригінальної міграції
- `warehouse_type TEXT` — додано в міграції #014

Це дублювання. Стару колонку `type` планували видалити (міграція #016), але CHECK constraint залишився.

**Рекомендація:** Видалити `type`, залишити `warehouse_type` з нормальним CHECK.

#### 2.3.2 🟡 parent_shop_id — заповнено не для всіх
14 цехів/складів без `parent_shop_id`. Без цього неможливо зрозуміти, до якого магазину належить цех для маршрутизації заявок.

**Рекомендація:** Заповнити `parent_shop_id` для всіх записів.

#### 2.3.3 🟡 Немає адресної логістики (зони, стелажі, полиці)
У реальному складі кожен товар має адресу зберігання (ряд-стелаж-полиця). Зараз цього немає.

**Рекомендація:** Додати `storage_locations` (зонування складу) та `product_storage_locations` — прив'язку товару до місця зберігання. Це критично для складу з ~160 позицій, які мають бути швидко знайдені.

## 3. Товарний довідник

### 3.1 Категорії

#### 3.1.1 ✅ Ієрархічні категорії
Дерево категорій через `parent_id` + `sort_order`. Коректно.

#### 3.1.2 🟡 Категорії російською мовою
З даних Poster: "Бытовая химия", "Упаковка", "Расходники для пекарні" тощо. Мова не відповідає політиці проєкту (українська).

**Рекомендація:** Перекласти назви категорій на українську.

### 3.2 Одиниці виміру

#### 3.2.1 🔴 Немає довідника одиниць виміру
Одиниця виміру — текстове поле з дефолтом 'шт'. Але в реалі:
- `шт` — штуки
- `уп` — упаковки
- `кг` — кілограми
- `л` — літри
- `м` — метри
- `пач` — пачки
- `ящ` — ящики
- `рул` — рулони
- `гр` — грами
- `мл` — мілілітри

Зараз одиниця просто текст — не можна нормально конвертувати (напр., "1 ящик = 12 упаковок"). При замовленні через Telegram плутанина: "2 уп" = 2 упаковки, а в системі товар може бути в "шт".

**Рекомендація:** Створити `units_of_measure` з:
- `id`, `code` ('шт', 'кг', 'л', 'уп', 'пач', 'ящ'), `name`, `category` (weight, volume, piece, pack)
- У `products` — `base_unit_id` (базова одиниця: шт, кг) та `purchase_unit_id` (одиниця закупівлі: ящик, упаковка)
- Додати `unit_conversions` — коефіцієнти конвертації (1 ящик = 12 шт; 1 упаковка = 100 шт)

Це прибере плутанину в Telegram-боті, коли "2 уп" ≠ "2 шт".

## 4. Інтеграції та бот

### 4.1 Telegram Bot

#### 4.1.1 ✅ Архітектура вебхуку — правильна
- Secret token перевірка
- Service role key для операцій запису
- Rate limiting
- Логування всіх повідомлень

#### 4.1.2 🔴 Пошук товарів за назвою — слабкий
Зараз використовується IDF-скорринг із prefix matching у коді. Це працює, але:
- Не знаходить "Ґрасаторе" коли в БД "Сграсаторе"
- Плутає "Рукавички S" / "Рукавички М" / "Рукавички L"
- Не розуміє "Пакети сміття великі" = "Пакет для сміття 120л"
- Чутливий до порядку слів

**Рекомендація:** Використовувати `pg_trgm` (вже встановлений) для пошуку в БД замість власного алгоритму:
```sql
SELECT id, name FROM products 
WHERE is_active 
  AND name % search_text  -- trigram similarity
ORDER BY similarity(name, search_text) DESC 
LIMIT 1;
```

Або комбінувати: спочатку ILIKE, потім pg_trgm, потім власний алгоритм.

#### 4.1.3 🟡 Немає підтвердження замовлення в групі
Зараз бот мовчки створює заявку. Користувач не знає, створилась вона чи ні. Через 5 хвилин може написати те саме ще раз.

**Рекомендація:** Додати реакцію ✅ на повідомлення (через `setMessageReaction` в Telegram API) або 👍/👎, щоб користувач бачив — заявка прийнята.

#### 4.1.4 🟡 Немає обробки new_chat_members
При додаванні бота в групу або приєднанні нового учасника Telegram надсилає `chat_member` update. Бот не обробляє це — код просто проходить далі.

**Рекомендація:** Обробляти `new_chat_members` — вітати, пояснювати як користуватися ботом.

### 4.2 Poster API

#### 4.2.1 🟡 Немає регулярної синхронізації з Poster
Товари, постачальники та склади імпортовані разово. Якщо в Poster з'явиться новий товар або зміниться ціна — CRM про це не дізнається.

**Рекомендація:** Додати фонове завдання (через pg_cron або окремий скрипт):
- Щоденна синхронізація товарів (нові, змінені)
- Щотижнева синхронізація цін
- Сповіщення про розбіжності

## 5. Безпека та надійність

### 5.1 🟡 Аудит — частковий
- Тригери логують INSERT/UPDATE/DELETE ✅
- Але немає `old_values` — якщо хтось змінив ціну, не видно було/стало
- audit_log не має retention policy

**Рекомендація:** Додати `old_values` JSONB у `audit_log`, налаштувати retention (90 днів).

### 5.2 🟡 Немає бекапів на рівні проєкту
БД на self-hosted Supabase, але немає:
- Регламенту бекапів
- Перевірки відновлення (restore test)
- Бекапів конфігурації Vercel (env vars)

**Рекомендація:** Налаштувати:
- Щоденний бекап БД (pg_dump)
- Зберігання бекапів 7 днів
- Тест відновлення хоча б раз на місяць

## 6. UX/UI для оператора

### 6.1 🟡 Немає сканера штрих-кодів
На складі ~160 позицій. При оприбуткуванні та інвентаризації оператор вводить назву товару вручну — повільно, з помилками.

**Рекомендація:** Додати інтеграцію зі сканером штрих-кодів (CS). Приймання: скануєш штрих-код → підставляється товар + остання ціна. Інвентаризація: скануєш → вводиш кількість.

### 6.2 🟡 Звіти тільки через дашборд
Немає:
- Excel-звіту "Залишки по складу" (є ExportButton, але не для всього)
- Звіту "Товари нижче min_stock" для постачальника
- Журналу рухів за період (є rpc_stock_movements_list, але в UI не виведено)
- Аналізу оборотності (які товари не рухаються 30/60/90 днів)

### 6.3 🟡 Немає офлайн-режиму
CRM залежить від Supabase. Якщо інтернет зникне — склад сліпий. Для складу критично мати можливість:
- Прийняти товар (паперова накладна, потім у систему)
- Провести інвентаризацію

**Рекомендація:** Додати друковану форму прибуткової накладної (PDF) — оператор друкує, підписує, потім вносить у систему. Інвентаризаційний опис теж.
